## はじめに

生成AIの活用は、ChatGPTから始まり、Claude、Claude Code、Codexなどを人間が都度呼び出し、指示を与えて結果を確認する使い方が主流でした。

この使い方は非常に強力ですが、一方で活用度合いが個人のスキルや習慣、もともとの業務理解に依存しやすいという課題があります。

生成AIをうまく使える人はどんどん効率化できますが、そうでない人にとっては「便利そうだけど、何を頼めばいいかわからない」となりがちです。

そのため、組織全体の業務改善につなげようとすると、単にAIツールを配るだけでは不十分です。

これから重要になるのは、人間が毎回AIを呼び出すのではなく、特定の環境にAIエージェントを常駐させ、会話や作業履歴を記憶し、繰り返し行う作業をスキルとして蓄積していくことだと考えています。

その文脈で注目しているのが、Hermes Agentです。

## Hermes Agentとは

Hermes Agentは、Nous Researchが開発しているオープンソースの自律型AIエージェントです。

公式サイトでは、2026年2月にリリースされたエージェントとして紹介されており、Linux、macOS、WSL2で動作します。

また、オープンソースかつMITライセンスで、テレメトリやデータ収集がなく、記憶データはローカルの `~/.hermes/` に保存されるとされています。

公式リンクはこちらです。

* [Hermes Agent — 記憶を持つオープンソースAIエージェント](https://hermes-agent.org/ja/)
* [NousResearch/hermes-agent](https://github.com/NousResearch/hermes-agent)

Hermes Agentは、単にLLMと会話するためのCLIツールではありません。

特徴として、次のような要素を持っています。

* セッションをまたいだ記憶
* 過去の会話や作業履歴の検索
* 作業経験からのスキル生成
* スキルの改善
* CLI、メッセージング、Cron、MCPなどを通じた外部連携
* ローカルまたはサーバー上での常駐実行

一言でいえば、Hermes Agentは「使うほど業務を理解してくれるAIエージェント」です。

一時的にAIへ質問するのではなく、継続的に作業を任せることで、業務の文脈や手順を蓄積していくことを目指したツールだと捉えています。

常駐型である点は、一時期話題になったOpenClawのような発想にも近いです。

## AIツールからAIエージェントへ

これまでの生成AI活用は、人間がAIに依頼する形が中心でした。

非常に便利ですが、基本的には「人間が作業を開始する」ことが前提です。

人間が課題を見つけ、AIに依頼し、出てきた結果を確認する。

つまり、AIは強力な補助者ではあるものの、仕事の流れ自体は人間が握っています。

一方で、Hermes Agentのような常駐型AIエージェントを使うと、発想が少し変わります。

AIが特定の環境に常駐し、Slackのようなメッセージングツール、定期実行、CLI、MCP、各種社内ツールなどと接続されることで、人間が毎回ゼロから依頼しなくても、業務の初動をAI側に任せられるようになります。

これは、以前書いた「Human in the Loop から Human on the Loop へ」という話ともつながります。

[AI活用を個人技で終わらせないために ②Human in the Loop から　Human on the Loopへ](https://lyricrime.com/posts/human-on-the-loop/)

人間が毎回AIに依頼して結果を見る状態から、AIが先に動き、人間は設計・監視・最終判断に寄る状態へ。

Hermes Agentは、そのような働き方を実現するための実験材料として面白いと思います。

## 基本的な仕組み

Hermes Agentの中心的な要素として、特に重要なのは次の3つです。

* メモリ
* スキル
* ツール連携

それぞれ見ていきます。

## メモリ

Hermes Agentは、セッションをまたいで情報を保持する永続的なメモリを持ちます。

これにより、毎回同じ前提を説明しなくても、過去の作業やユーザーの好みを参照しながら動けるようになります。

通常のChatGPTやClaudeでも、会話の中では文脈を保持できます。

しかし、会話をまたいだり、別の作業として扱ったりすると、前提をもう一度説明する必要があります。

Hermes Agentでは、作業履歴や判断の文脈を継続的に蓄積していくことで、より業務に寄り添った動きが期待できます。

たとえば、組織で使う場合は、次のような情報を覚えさせる使い方が考えられます。

* よく使う調査観点
* チーム固有のレビュー観点
* 業務ツールの使い方
* 定例的な報告フォーマット
* 過去に判断した方針
* 問い合わせ対応時の確認手順
* 障害調査時に見るべきログやメトリクス

このような情報が記憶されていくと、AIは単なる文章生成ツールではなく、業務の前提を知っているアシスタントに近づいていきます。

## スキル

Hermes Agentには、スキルという概念があります。

スキルは、特定の作業手順やプロンプト、ツール利用をまとめた再利用可能な単位です。

また、AnthropicのAgent Skill形式での保存が可能とされており、Hermes Agentの中だけに閉じない形でスキルを扱える点も面白いです。

これは、将来的に別のAIエージェント環境へ乗り換えたり、複数のツール間で作業手順を再利用したりする可能性を示しています。

開発や業務改善の文脈で考えると、たとえば次のようなスキルが考えられます。

* 変更内容を読み、QA観点を洗い出すスキル
* メッセージングツール上の依頼文から必要な確認事項を抽出するスキル
* 技術調査を行い、要点・判断材料・リスクをまとめるスキル
* 障害や問い合わせの初動整理を行うスキル
* 定例レポートの材料を集めるスキル
* 議事録から決定事項と次アクションを抽出するスキル
* ドキュメントの更新候補を洗い出すスキル

重要なのは、これらが一度きりのプロンプトではなく、改善されていく作業手順として扱える点です。

AIに毎回同じ説明をするのではなく、「この作業はこう進める」という型を蓄積していく。

この考え方は、個人のAI活用を組織の資産に変えていくうえで重要だと思います。

## ツール連携

Hermes Agentは、ファイル操作、検索、シェル実行、ブラウザ操作、コード実行などのツールを呼び出せます。

また、MCP、Cron、メッセージングツール、CLIなどと組み合わせることで、外部システムと連携した作業も可能になります。

ツールを使えることで、AIは単に文章を生成するだけでなく、実際にファイルを読んだり、コマンドを実行したり、情報を取得したりできます。

たとえば「障害や問い合わせの初動整理を行うスキル」を考える場合、エージェントは次のようなツールにアクセスできると便利です。

* エラー監視ツール
* ログ分析ツール
* メトリクス監視ツール
* チケット管理ツール
* リポジトリ管理ツール
* ドキュメント検索ツール
* メッセージングツール

人間が毎回これらのツールを開いて、関連情報を探し、状況を整理する代わりに、Hermes Agentが初動の情報収集を担います。

もちろん、最初から自動で判断・返信・修正まで任せるのは危険です。

特に、本番環境に影響する操作や、外部に返信する処理、データを書き換える処理は慎重に扱う必要があります。

そのため、最初は読み取り専用のツール連携から始めるのが現実的です。

たとえば、エラー監視ツールやログ分析ツールから情報を取得し、状況整理と返信案の作成までを行う。

この範囲であれば、人間の判断を残しつつ、初動調査の負担をかなり下げられます。

## インストール

インストールは、READMEに従って進めます。

Linux、macOSが推奨されており、Windowsの場合はWSL2が必要です。

```sh
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash
```

組織で利用する場合は、どの端末、どの環境で動かすかを先に検討した方がよいです。

Hermes Agentは、LLMそのものというより、AIエージェントを動かすためのハーネスに近いツールです。

そのため、実際にどのモデルやAPIを利用して動かすかを決める必要があります。

選択肢としては、次のようなものが考えられます。

* OpenAI API
* Nous Portal
* OpenRouter
* ローカルLLM
* その他のLLM API

個人検証であれば手元のAIツールと組み合わせてもよいですが、チームや組織で利用する場合は、利用規約やデータの扱いを確認したうえで、API利用を前提に考える方が安全です。

## Dockerの2つの使い方

Hermes AgentにおけるDockerの使い方は、大きく2種類あります。

1つ目は、Hermes Agent自体をDockerコンテナ内で動かす構成です。

この場合、ユーザーデータ、設定、APIキー、セッション、スキル、メモリをホスト側のディレクトリから `/opt/data` にマウントし、イメージ自体はステートレスに保ちます。

```bash
mkdir -p ~/.hermes

docker run -it --rm \
  -v ~/.hermes:/opt/data \
  nousresearch/hermes-agent setup
```

セットアップ後、メッセージングツールなどのGatewayとして常駐させる場合は、次のように起動できます。

```bash
docker run -d \
  --name hermes \
  --restart unless-stopped \
  -v ~/.hermes:/opt/data \
  -p 8642:8642 \
  nousresearch/hermes-agent gateway run
```

2つ目は、Hermes Agent本体はホストOSで動かし、エージェントが実行するコマンドだけをDockerサンドボックス内で動かす構成です。

この場合、Hermes Agentの操作性はホスト側に残しつつ、ファイル操作やコマンド実行の影響範囲をDocker内に閉じ込めることができます。

整理すると、次のようになります。

| 使い方                        | 内容                              | 向いているケース                       |
| -------------------------- | ------------------------------- | ------------------------------ |
| Hermes Agent自体をDockerで動かす  | GatewayやDashboardも含めてコンテナ常駐させる  | VPSやクラウドVMなどに常駐させたい場合          |
| Terminal backendをDockerにする | Hermes本体はホスト、コマンド実行だけDocker内で行う | ローカル検証で安全にファイル操作やコマンド実行をさせたい場合 |

注意点として、Claude CodeやCodexのようなツールは、起動した場所をワークスペースとして扱うことが多いです。

一方、Hermes AgentをDockerコンテナ内で動かす場合、マウントされているボリューム全体が見えることになります。

このため、どのディレクトリをマウントするか、どの権限で動かすかは慎重に設計した方がよいです。

## 活用例：メッセージングツールでの作業依頼の一次対応

まず試したい活用例は、メッセージングツールで来る作業依頼の一次対応です。

リモートワークが可能な仕事では、多くの作業トリガーがメッセージングツール上にあります。

たとえば、次のような依頼です。

* エラー通知の内容を確認してほしい
* ログを調査してほしい
* 変更内容をレビューしてほしい
* エラーや権限不足の原因を調べてほしい
* チケットや問い合わせの内容を整理してほしい
* 過去の類似対応を探してほしい

これらは人間が対応すべき判断を含む一方で、毎回ゼロからすべての関連ツールにアクセスして内容を読む必要があるわけではありません。

Hermes Agentに任せる場合、まずは次のような流れが考えられます。

1. メッセージングツールでメンションされた依頼を受け取る
2. 依頼内容を分類する
3. 不足している情報を洗い出す
4. 参照すべきドキュメントや過去ログを探す
5. 必要に応じて監視ツールやログ分析ツールから情報を取得する
6. 状況を整理する
7. 返信案を作成する
8. 人間が確認して送信する

最初から自動返信させるのではなく、まずは「返信案の作成」までに留めるのが安全です。

この形であれば、人間は毎回依頼内容を読み解くところから始める必要がなくなります。

Hermes Agentが初動整理を行い、人間は判断や修正に集中できます。

## 業務の棚卸しをしなくても始められる

この使い方の良い点は、AI活用でよくある「業務の棚卸し」をしなくても始められることです。

通常、業務にAIを適用しようとすると、次のような手順が必要になります。

1. 自分の業務を洗い出す
2. AIで代替できそうな作業を探す
3. 作業手順を言語化する
4. プロンプトを作る
5. 実際に運用して改善する

これは正攻法ですが、特に日々忙しいチームでは「AI活用できそうな業務を洗い出しましょう」と言われても、それ自体が負担になります。

一方で、Hermes Agentをメッセージングツールと組み合わせると、少し違う始め方ができます。

依頼が来たタイミングで、まずHermes Agentに一度対応させる。

その対応がうまくいったら、「今回の手順を覚えておいて」と指示する。

これを繰り返すことで、実際に発生した業務をベースにスキルが蓄積されていきます。

つまり、事前に大きな棚卸しをしなくても、実際の仕事の流れの中で業務手順をAIエージェントに移していけます。

これは、個人のAI活用を組織の業務改善につなげるうえで、かなり現実的なアプローチだと思います。

## 最初にやるべきこと

Hermes Agentをいきなり本格導入するのではなく、まずは小さく検証するのがよさそうです。

おすすめは、次のような順番です。

### 1. 読み取り専用の作業から始める

最初は、情報取得や要約に限定します。

たとえば、次のような作業です。

* エラー通知の要約
* ログ調査の初動整理
* チケット内容の整理
* 変更内容の要約
* ドキュメント検索
* 返信案の作成

書き込みや自動実行を伴う作業は、後回しにします。

### 2. 人間の確認を必ず挟む

最初から自動返信や自動修正をさせるのではなく、人間の確認を必ず挟みます。

AIエージェントが作るのは、あくまで「判断材料」や「返信案」です。

最終判断は人間が行います。

この形であれば、AIのミスを前提にしながらも、初動の負担を減らせます。

### 3. うまくいった対応をスキル化する

一度うまくいった対応は、スキルとして保存します。

* エラー通知を受け取ったときの初動確認手順
* 問い合わせを整理するときの観点
* 変更内容からQA観点を洗い出す手順
* 定例報告の作成手順
* 技術調査のまとめ方

これを繰り返すことで、AIエージェントが徐々に業務に適応していきます。

## 注意点

### 1. 権限を与えすぎない

AIエージェントに強い権限を与えすぎると、ミスをしたときの影響が大きくなります。

最初は読み取り専用に近い権限から始め、必要な範囲だけを許可するべきです。

### 2. 記憶させる情報を選ぶ

メモリがあることは強みですが、何でも記憶させればよいわけではありません。

機密情報、個人情報、認証情報、不要になった古い判断などを不用意に残すと、後から問題になる可能性があります。

記憶させる情報の粒度や削除方針は、運用前に考えておいた方がよいです。

## まとめ

Hermes Agentは、単なるAIチャットツールではなく、業務の文脈を記憶し、作業手順をスキルとして蓄積し、外部ツールと連携しながら動く常駐型AIエージェントです。

これまでの生成AI活用は、人間が毎回AIを呼び出して使う形が中心でした。

しかし、これからはAIエージェントが特定の環境に常駐し、業務の初動を担い、人間は設計・監視・最終判断に寄っていく流れが強まると考えています。

特に、メッセージングツール上の依頼対応、エラー調査、ログ確認、変更内容の整理、問い合わせの初動対応のような業務は、Hermes Agentの検証対象として相性がよさそうです。

最初から完全自動化を目指す必要はありません。

まずは、読み取り専用の情報収集と返信案の作成から始める。うまくいった対応をスキル化する。

実際の依頼を通じて、少しずつAIエージェントに業務の流れを覚えさせる。

このような進め方であれば、AI活用を個人技で終わらせず、組織の業務改善につなげやすくなるのではないかと思います。

Hermes Agentはまだ検証段階のツールではありますが、「AIツールを使う」から「AIエージェントが業務に常駐する」への変化を考えるうえで、とても面白い存在です。
